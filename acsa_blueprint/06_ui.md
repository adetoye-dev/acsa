# Phase 6: Visual Workflow Editor (UI)

The user interface lets developers and non‑technical users compose workflows
visually and manage executions.  In line with Acsa’s local‑first philosophy,
the UI runs as a separate process that reads and writes YAML files rather
than storing workflows in a database.  This phase outlines how to build the
initial editor using React and Next.js.

## 1. Project Initialization

1. **Create the app.**  Within the `ui/` directory, run the Next.js
   initialization command (`npx create-next-app@latest`) with TypeScript
   support.  Choose the “app router” if available.

2. **Clean up boilerplate.**  Remove default pages, CSS files, and API
   examples.  Keep only `pages/index.tsx` (or `app/page.tsx` if using the
   app router) and a global CSS file (preferably Tailwind CSS or your
   framework of choice).

3. **Install dependencies.**  Use your package manager (npm, pnpm, or yarn) to
   install:

   - `react-flow-renderer` for the node‑based canvas.
   - `yaml` for converting YAML to/from JavaScript objects.
   - `axios` for HTTP requests (later used to interact with the engine’s API).
   - `tailwindcss` and `@tailwindcss/forms` for styling.  Configure Tailwind
     with a basic palette that matches your brand colors.

## 2. Core UI Components

1. **Workflow canvas.**  Use `ReactFlow` to display nodes and edges.
   Implement node types corresponding to built‑in nodes (HTTP request, LLM,
   condition, etc.).  Each node should be a draggable component with a label
   and icons representing its type.  Edges should be drawn automatically
   when connecting nodes.

2. **Node inspector panel.**  When a node is selected on the canvas, display
   a side panel with form fields corresponding to the node’s parameters
   (e.g., URL and method for an HTTP node).  Use controlled components and
   update the underlying YAML structure on change.  Provide validation and
   tooltips.

3. **Workflow explorer.**  A sidebar listing all workflows found in the
   `workflows/` folder.  Allow users to create a new workflow, duplicate an
   existing one, or delete one.  Highlight the currently open workflow.

4. **Top bar actions.**  Include actions such as save (write YAML file), run
   workflow (manual trigger), and view execution history (links to
   observability pages).  For the save action, convert the canvas state to
   YAML and write the file to `workflows/{name}.yaml`.  Use the File System
   Access API or send a POST request to the engine if running in a server
   context.

## 3. Data Flow

1. **YAML as single source of truth.**  Maintain the workflow state in
   memory as a JavaScript object.  When the user loads a workflow, parse its
   YAML into the object.  When they save, serialize the object back to YAML.
   Keep the object consistent with the canvas state at all times.

2. **Hot reload.**  Optionally implement a file watcher in the engine (or
   Next.js API route) that notifies the UI when YAML files change on disk.
   This enables collaboration when multiple editors are running.

3. **API integration.**  Provide minimal endpoints in the engine (e.g.,
   `/api/workflows`) that the UI can call to list workflows, read YAML,
   write YAML, and start executions.  For local‑first mode, allow the UI to
   read/write files directly using the browser’s file system APIs.  In
   hosted mode, route these actions through the engine’s REST API.

## 4. Security Considerations

1. **No remote code execution.**  Never interpret user‑provided content as
   code in the browser.  Node parameter fields should be sanitized to avoid
   injection attacks.  Use strict Content Security Policies (CSP) and avoid
   `dangerouslySetInnerHTML` unless absolutely necessary.

2. **Access control.**  If the UI is hosted, implement authentication and
   authorization.  For local‑first mode, rely on the user’s local file
   permissions.  Avoid exposing workflow files over HTTP without
   authorization.

3. **Secret management.**  Do not allow secrets to be entered directly into
   the workflow YAML from the UI.  Provide a mechanism to reference
   environment variables or secret keys stored on the server.  Display
   placeholders in the UI and document how to configure secrets.

4. **Error handling.**  Catch and display validation errors when saving
   workflows.  Prevent invalid YAML from being written to disk.  Use
   optimistic UI updates but roll back on failure.

## 5. Future Enhancements (Out of Scope for MVP)

1. **Real‑time collaboration.**  Integrate WebSockets or WebRTC to allow
   multiple users to edit the same workflow simultaneously.

2. **Templating and component library.**  Provide reusable subflow
   components that users can drag onto the canvas to encapsulate common
   patterns.

3. **Dark mode and themes.**  Support theme switching and custom branding.

Completing this phase will give users a minimal but functional way to build
workflows visually, save them as YAML, and trigger executions.  Keep the UI
simple and avoid over‑engineering; focus on establishing a reliable
foundation.
